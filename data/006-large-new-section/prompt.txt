Add a new section called "Tools for the agent" with the following contents:

Agent Tools - Detailed Descriptions
1. read_file_tool
Purpose: Retrieve the complete text content of any file within the project directory.

Parameters:

file_path (string, required): A relative path from the project root to the target file (e.g., main.tex, sections/introduction.tex, refs.bib).
Behavior:

The tool constructs the full path by joining the project root with the provided relative path.
First checks if the file exists at the computed path. If not, returns an error message indicating the file does not exist.
Verifies that the path points to a file (not a directory). If it's a directory, returns an error.
Reads the entire file content as text using UTF-8 encoding by default.
Returns the complete file content on success.
Catches and reports any I/O exceptions (permissions, encoding errors, etc.) with a descriptive error message.
Return Value:

On success: The complete text content of the file.
On failure: An error string describing why the read failed (e.g., "File does not exist", "Not a file", "Permission denied").
Safety Notes:

Paths are used directly without additional validation (contrast with delete_file_tool and rename_file_tool, which use _resolved_under_root for stricter validation).
Can read any file type, but is primarily used for LaTeX (*.tex), bibliography (*.bib), and text-based project files.
Common Use Cases:

Reading main.tex to understand the document structure.
Inspecting refs.bib to see existing bibliography entries.
Reading section files (e.g., sections/introduction.tex) to understand content before edits.
2. edit_file_tool
Purpose: Create a new file or completely replace the contents of an existing file in the project directory.

Parameters:

file_path (string, required): A relative path from the project root to the target file (e.g., sections/new_section.tex, main.tex).
content (string, required): The complete, final content to write to the file.
Behavior:

Constructs the full path by joining the project root with the provided relative path.
Automatically creates any missing parent directories using mkdir(parents=True, exist_ok=True). This enables the tool to create nested directory structures if needed.
Writes the provided content entirely to the file, replacing any existing content (if the file already exists).
Uses text mode with UTF-8 encoding.
Returns a success message on completion.
Catches and reports any I/O exceptions (permissions, disk full, encoding errors) with a descriptive message.
Return Value:

On success: "Successfully updated '{file_path}'."
On failure: An error string (e.g., "Permission denied", "Disk full", "Invalid path").
Important Notes:

This is a complete replacement operation. The agent must provide the full, final content of the file—not just the lines to change. This prevents accidental data loss and ensures the agent has explicit control over the final state.
Parent directories are created automatically, simplifying workflows where the agent creates new sections in nested locations.
Common Use Cases:

Creating new section files (e.g., writing sections/methodology.tex with complete content).
Updating main.tex with new \input statements to include sections.
Modifying refs.bib to add or update bibliography entries.
Creating LaTeX preamble files or configuration files.
3. delete_file_tool
Purpose: Permanently remove a file from the project directory.

Parameters:

file_path (string, required): A relative path from the project root to the file to delete.
Behavior:

Uses _resolved_under_root() to validate the path and ensure it stays within the project directory. This function resolves the path to its canonical form and verifies that the final path is still under the project root—this prevents path traversal attacks (e.g., using ../../../sensitive_file).
If the path attempts to escape the project root, raises a ValueError with message "Path escapes project root".
Checks if the file exists. If not, returns an error indicating non-existence.
Verifies that the path is a file (not a directory). Directories cannot be deleted with this tool.
Deletes the file using path.unlink().
Returns a success message on completion.
Catches ValueError (from path validation) and reports it directly. Catches other exceptions and reports them with a descriptive message.
Return Value:

On success: "Successfully deleted '{file_path}'."
On failure: An error string (e.g., "Path escapes project root", "File does not exist", "Is a directory", "Permission denied").
Safety Notes:

Path validation via _resolved_under_root() is strict and prevents malicious or accidental attempts to delete files outside the project.
Only deletes individual files, never directories or recursive structures.
No recycle bin or undo: deletions are permanent.
Common Use Cases:

Removing outdated section files when reorganizing the document.
Deleting temporary or draft files.
Cleaning up figures that are no longer used.
4. rename_file_tool
Purpose: Rename a file or move it to a different location within the project directory (supports both operations since rename is a move).

Parameters:

from_path (string, required): The current relative path of the file from the project root.
to_path (string, required): The new relative path for the file from the project root.
Behavior:

Uses _resolved_under_root() to validate both source and destination paths, ensuring neither escapes the project root.
If either path attempts to escape the project root, raises a ValueError with appropriate messaging.
Verifies that the source file exists and is actually a file (not a directory).
Checks if the source and destination paths are identical. If so, returns a no-op message.
Checks if a file already exists at the destination path. If it does, returns an error to prevent overwriting.
Creates the destination parent directories if they don't exist using mkdir(parents=True, exist_ok=True).
Performs the rename operation using src.rename(dst), which atomically moves the file.
Returns a success message on completion.
Catches and reports exceptions with descriptive messages.
Return Value:

On success: "Successfully renamed '{from_path}' to '{to_path}'."
On failure: An error string (e.g., "Path escapes project root", "Source file does not exist", "Destination already exists", "Permission denied").
Safety Notes:

Strict path validation prevents escaping the project directory.
Does not overwrite existing files; returns an error if the destination already exists.
Creating destination parent directories automatically supports reorganizing documents into new directory structures.
Common Use Cases:

Renaming sections as the document structure evolves (e.g., sections/background.tex → sections/related_work.tex).
Moving files to subdirectories to organize content (e.g., section1.tex → chapters/chapter1/section1.tex).
Note: The agent's system prompt mentions updating references after rename (e.g., updating \input statements in main.tex), but this tool only moves the file—the agent must handle reference updates as a separate step.
5. list_files_tool
Purpose: Browse the project directory structure to understand what files are present.

Parameters:

recursive (boolean, optional, default=True): If True, lists all files in the project directory and subdirectories recursively. If False, lists only files in the root directory (not subdirectories).
Behavior:

If recursive=True: Uses folder_path.rglob("*") to recursively find all items, filters to only files (not directories), and collects their relative paths from the project root.
If recursive=False: Uses folder_path.iterdir() to list only top-level items, filters to files, and collects their relative paths.
Sorts the collected file paths alphabetically for consistent, readable output.
Formats the output as a bulleted list with each file path on a new line.
Returns a message indicating "No files found" if the directory is empty or contains only subdirectories.
Catches and reports any I/O exceptions.
Return Value:

On success: A formatted string like:
Files in project directory:
  - main.tex
  - refs.bib
  - sections/introduction.tex
  - sections/methodology.tex
  - figures/fig1.png
On empty directory: "No files found in the project directory."
On error: An error message describing the issue.
Behavior Notes:

Listing is always non-destructive and read-only.
The recursive parameter is useful: recursive listing gives a full project overview, while non-recursive listing shows only top-level files (useful for quick checks of the root structure).
Files are sorted alphabetically within the list for predictable output.
Common Use Cases:

First step in any agent request: list files to understand the project structure.
Verifying that a file was created or deleted as expected.
Confirming the presence of expected directories like sections/, figures/, chapters/.
Planning document reorganization by reviewing the current structure.
6. compile_latex_tool
Purpose: Compile the LaTeX project to generate a PDF and report any compilation errors.

Parameters: None. This tool takes no parameters.

Behavior:

Searches for an available LaTeX compiler in this order: tectonic, pdflatex, latexmk. Uses the first one found.
If no compiler is found, returns an error message: "Error: No LaTeX compiler found (tried tectonic, pdflatex, latexmk)."
Based on the detected compiler, constructs the appropriate command:
Tectonic: ["tectonic", "-X", "compile", "main.tex"]
latexmk: ["latexmk", "-pdf", "-interaction=nonstopmode", "main.tex"]
pdflatex: ["pdflatex", "-interaction=nonstopmode", "main.tex"]
Executes the command with:
Working directory set to the project folder.
Output captured (both stdout and stderr).
Text mode (not binary).
A 120-second timeout to prevent runaway compilation.
If compilation succeeds (return code 0), returns: "SUCCESS"
If compilation fails (non-zero return code), returns: "FAILED: {stderr_output}" with the error messages from the compiler.
If compilation times out (exceeds 120 seconds), returns: "Error: Compilation timed out."
Catches and reports any other exceptions.
Return Value:

On success: "SUCCESS"
On failure: "FAILED: {error_details}" where error_details are compiler error messages.
On timeout: "Error: Compilation timed out."
On missing compiler: "Error: No LaTeX compiler found (tried tectonic, pdflatex, latexmk)."
Important Notes:

Always compiles main.tex (assumes this is the root document).
All three compilers are tried in preference order (Tectonic preferred, then pdflatex, then latexmk) based on what's available on the system.
The -interaction=nonstopmode flag for pdflatex and latexmk ensures compilation continues even on errors, returning all error messages.
The 120-second timeout prevents infinite loops in the build system.
Returns only a status string; the actual PDF file is generated on disk and must be retrieved separately.
Common Use Cases:

Validating LaTeX syntax after edits (e.g., checking if a new section was formatted correctly).
Detecting errors introduced by the agent's modifications (missing braces, undefined references, etc.).
Providing feedback to the agent for error recovery—the agent can read error messages and attempt fixes.
Limitations:

The 120-second timeout may be insufficient for very large documents with many cross-references.
Does not return the PDF content directly; the frontend must retrieve the PDF separately.
Error messages are compiler-specific; the agent must interpret them.
7. move_attached_image_to_project_tool
Purpose: Move a user-uploaded image file into the project's figures/ directory so it can be referenced and included in the LaTeX document.

Parameters: None. This tool operates on the currently attached image (provided at initialization time).

Behavior:

Checks if an image is currently attached. If not, returns an error: "Error: No image is currently attached."
Constructs the path to the figures/ directory within the project.
Creates the figures/ directory if it doesn't exist using mkdir(parents=True, exist_ok=True).
Computes the destination path by combining the figures directory with the image's original filename (e.g., figures/photo.png).
Implements automatic collision handling: if a file already exists at the destination path, the tool incrementally appends a counter to the filename (e.g., photo-1.png, photo-2.png, etc.) until a unique name is found.
Moves (not copies) the image file from its temporary upload location to the final destination in the figures directory using shutil.move().
Returns a success message with the relative path where the image was moved to.
Catches and reports any exceptions.
Return Value:

On success: "Moved attached image to '{relative_path}'."
Example: "Moved attached image to 'figures/diagram.png'."
On failure: An error string (e.g., "No image is currently attached", "Permission denied", "Disk full").
Collision Handling Example:

User uploads diagram.png → stored as figures/diagram.png.
User uploads diagram.png again → stored as figures/diagram-1.png (to avoid overwriting).
User uploads diagram.png again → stored as figures/diagram-2.png.
Important Notes:

The image is moved (not copied), so it's removed from its temporary location.
The tool preserves the original filename extension and base name where possible, only appending a counter if necessary.
The figures/ directory is created automatically if it doesn't exist.
The image must have been uploaded and attached before this tool can be called; it cannot operate on arbitrary files.
Common Use Cases:

As the final step in an image insertion workflow: user uploads image → tool moves it to figures/ → agent inserts \includegraphics reference in the document.
Handling user uploads of diagrams, plots, photographs, or screenshots to include in documents.
Organizing all project images in a centralized figures/ directory to maintain project structure.
Workflow Integration:

Frontend uploads image and stores its temporary path.
Frontend passes the image path to the Agent in the chat context.
Agent decides to include the image and calls move_attached_image_to_project_tool.
Agent then calls edit_file_tool to insert the LaTeX figure environment with \includegraphics{figures/image.png}.
Agent calls compile_latex_tool to verify the LaTeX is valid.
